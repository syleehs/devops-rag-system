# ADR-011: Automated CIDR Range Documentation via Pipeline-Generated PRs

## Status
Accepted

## Context

IBM Cloud Pipeline's migration from bare-metal to VPC Gen2 introduced dynamic IP allocation. With Kubernetes autoscaling enabled, nodes were created and destroyed based on demand. Each new node received a new IP address from the VPC's CIDR range.

This created a documentation problem:
- Internal network documentation listed specific IP ranges for cluster nodes
- Documentation was maintained manually by designated "document focals"
- As autoscaling created and destroyed nodes, actual CIDR ranges changed
- Manual documentation updates were slow, inconsistent, and often outdated
- Outdated IP documentation caused issues for:
  - Network security team configuring firewall rules
  - Support engineers diagnosing connectivity issues
  - Compliance audits requiring accurate network topology documentation
  - Cross-team communication about network boundaries

The document focals responsible for maintaining this documentation had no automated way to know when CIDR ranges changed.

## Decision

We built an automated pipeline that detected CIDR range changes from VPC autoscaling and generated pull requests to update internal network documentation automatically.

### 1. CIDR Range Detection

A scheduled pipeline job:
- Queried IBM Cloud VPC API for current CIDR ranges across all regions
- Compared current ranges against last known state stored in repository
- Detected additions, removals, and modifications to CIDR blocks

### 2. Automated PR Generation

When changes detected:
- Pipeline generated updated documentation with new CIDR ranges
- Automatically created a pull request against the internal docs repository
- PR included:
  - Summary of what changed (which regions, which ranges)
  - Before/after comparison of CIDR blocks
  - Timestamp of detection
  - Link to VPC console for verification

### 3. Document Focal Review

Document focals (designated reviewers) received PR notification:
- Reviewed the auto-generated changes
- Verified against VPC console if needed
- Approved and merged PR to update official documentation
- No manual CIDR detection or documentation writing required

### 4. Scheduling

Pipeline ran on schedule aligned with autoscaling patterns:
- Triggered more frequently during known high-traffic periods
- Daily baseline scan to catch any missed changes
- Triggered immediately after known scaling events

## Consequences

### Benefits
- ✅ Documentation always reflects current VPC state
- ✅ Document focals review rather than research changes
- ✅ Audit trail of all CIDR changes via Git PR history
- ✅ Eliminated manual CIDR monitoring task
- ✅ Network security team had reliable IP documentation
- ✅ Reduced time from CIDR change to documented change from days to hours

### Trade-offs and Risks

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Pipeline failure | CIDR changes go undetected | Pipeline alerting, daily fallback scan |
| Reviewers merge without checking | Incorrect documentation approved | Clear PR description, easy VPC console link |
| API rate limits | VPC API queries throttled | Respectful polling intervals, caching |
| Large CIDR changes | PR diff hard to review | Structured diff format, region-by-region breakdown |

## Implementation Details

### Pipeline Structure

```
Schedule trigger (every 4 hours)
  → Query VPC API for all regions
  → Compare against stored CIDR state
  → If changes detected:
      → Generate updated documentation
      → Create PR with diff
      → Notify document focals
  → Update stored CIDR state
  → Log scan completion
```

### VPC API Query (Pseudocode)

```bash
#!/bin/bash
# Fetch current CIDR ranges from VPC

REGIONS=("us-south" "eu-gb" "ap-north" "us-east")

for REGION in "${REGIONS[@]}"; do
  # Query VPC subnets for region
  ibmcloud is subnets \
    --resource-group-name cloud-pipeline \
    --region $REGION \
    --output json | \
    jq -r '.subnets[] | "\(.name): \(.ipv4_cidr_block)"'
done
```

### PR Generation

```bash
# Generate documentation update
generate_cidr_docs() {
  CURRENT_CIDRS=$1
  
  cat > docs/network/cidr-ranges.md << EOF
# VPC CIDR Ranges

*Auto-generated by CIDR monitoring pipeline*
*Last updated: $(date -u +%Y-%m-%dT%H:%M:%SZ)*
*Next scheduled scan: $(date -u -d "+4 hours" +%Y-%m-%dT%H:%M:%SZ)*

## Current CIDR Allocations

$CURRENT_CIDRS

## Change History

See Git history for full change log.
EOF
}

# Create PR
create_documentation_pr() {
  BRANCH="auto/cidr-update-$(date +%Y%m%d-%H%M%S)"
  
  git checkout -b $BRANCH
  git add docs/network/cidr-ranges.md
  git commit -m "auto: update CIDR ranges [$(date -u +%Y-%m-%d)]

Changes detected in VPC CIDR allocation:
$CHANGE_SUMMARY

Regions affected: $AFFECTED_REGIONS
Triggered by: scheduled CIDR monitoring pipeline"
  
  git push origin $BRANCH
  
  # Create PR via GitHub/GitLab API
  create_pr \
    --title "auto: CIDR range update $(date +%Y-%m-%d)" \
    --body "$PR_BODY" \
    --reviewers "$DOCUMENT_FOCALS" \
    --labels "auto-generated,network,documentation"
}
```

### PR Description Template

```markdown
## Automated CIDR Range Update

**Detected changes:** $(date -u)
**Regions affected:** us-south, eu-gb

### Changes

#### us-south
Added:
- 10.240.64.0/26 (new autoscaling nodes - us-south-3)

Removed:
- 10.240.32.128/26 (decommissioned nodes)

#### eu-gb
No changes.

### Verification

To verify: [IBM Cloud VPC Console - Subnets](https://cloud.ibm.com/vpc-ext/network/subnets)

### Action Required

Please review and merge if accurate. If discrepancies found, 
please comment before merging.
```

## Monitoring

Key metrics:
- `cidr_scan_success_total` - Successful scans (alert if stops incrementing)
- `cidr_changes_detected_total` - Changes found per scan
- `pr_creation_success_total` - PRs successfully created
- `pr_merge_latency_hours` - Time from PR creation to merge (track responsiveness)

## Lessons Learned

1. **Dynamic infrastructure requires dynamic documentation** - Static docs become wrong the moment they're written in an autoscaling environment
2. **Generate PRs, not direct commits** - Human review catches pipeline errors before they corrupt documentation
3. **Make review easy** - Link directly to VPC console in PR so reviewers can verify in one click
4. **Clear PR descriptions drive fast reviews** - Vague PRs sit unmerged; specific PRs get merged quickly
5. **Schedule matters** - Scan frequency should match infrastructure change frequency
6. **Audit trail is a free benefit** - Git PR history gives you complete CIDR change history automatically
